#+TITLE: Step Execution State Machine
#+AUTHOR: Darwin D. Wu
#+DATE: 2026-04-16
#+STATUS: Draft

* Goal
- Replace the current simple generator behavior with a spec-aligned step execution state machine.

* Scope
- Root discovery vs targeted step execution
- `StepRun`
- `StepPlanned`
- `StepError`
- `StepNotFound`

* Checklist
- [ ] Distinguish root execution (`stepId=step`) from targeted step execution (`stepId!=step`)
- [ ] Prevent targeted step executions from discovering or executing unrelated new steps
- [ ] Add explicit support for planning a step without immediately running it
- [ ] Add explicit support for returning `StepPlanned`
- [ ] Add explicit support for returning `StepRun`
- [ ] Add explicit support for returning `StepError`
- [ ] Add explicit support for returning `StepNotFound`
- [ ] Ensure immediate execution only happens when the spec allows it
- [ ] Ensure targeted executions search for the requested step while memoizing prior state
- [ ] Preserve deterministic behavior when code has not changed
- [ ] Add tests for root discovery flow
- [ ] Add tests for targeted step execution flow
- [ ] Add tests for missing-target-step behavior

* Exit Criteria
- [ ] The SDK can handle both discovery and targeted step execution correctly
- [ ] Step opcodes are emitted in the correct cases
- [ ] The runtime no longer relies on the current one-path flow control shortcut

* Out of Scope
- Parallel discovery
- Stack recovery for changed functions
