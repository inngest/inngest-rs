* Goal
- Bring call-request parsing, step execution, stack handling, and trimmed-payload hydration into one spec-tracked milestone.

* Scope

* Parallelism Design
- The first Rust-native parallelism surface will be `inngest::group::parallel!`.
- The macro will be the public API because it matches Rust expectations for heterogeneous async branches better than `Vec`-based callback APIs.
- The macro takes the execution helper explicitly as `parallel!(step => branch_a, branch_b, ...)`.
- The initial implementation targets the default "all" behavior only.
- `race!` remains a follow-up item and is not required to complete this milestone.

* Parallelism Execution Model
- `group::parallel!` should evaluate each branch sequentially within the SDK while treating the discovered Steps as logically parallel.
- The runtime should aggregate branch-local step-generator interruptions instead of aborting the entire request on the first unresolved Step.
- Targeted execution inside a parallel group must skip unrelated branches and continue searching before returning `StepNotFound`.
- Root discovery within `group::parallel!` must support returning multiple new Steps in a single `206` response.
- Once multiple Steps are reported from one request, subsequent requests must continue honoring `ctx.disable_immediate_execution=true`.
- The first implementation should prefer minimal control-flow changes over a full executor rewrite.

* Implementation Notes
- Add a new public `group` module and export `parallel!` from the crate root.
- Keep the branch aggregation machinery internal to the step runtime.
- Extend the step runtime state so it can distinguish branch-local target misses from final request-level `StepNotFound`.
- Reuse the existing opcode vector response shape for multi-step responses.
- Defer `race!`, explicit `parallelMode` metadata, and any broader generalized multi-step discovery outside `group::parallel!`.
